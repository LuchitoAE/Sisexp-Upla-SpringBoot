SECCIÓN 2 — PRUEBAS DE INTEGRACIÓN
═══════════════════════════════════════════

Título: "Suite de Pruebas — Validación de Comunicación"

Prueba 1 — Flujo Completo (Happy Path):
  Herramienta: Postman Collection "SISEXP Microservicios"
  Pasos:
    1. POST /api/auth/login → 200 {token: "eyJ..."}
    2. POST /api/techos-presupuestales → 201 {id:1, año:2026, montoTotal:115000}
    3. POST /api/actividades-poi/techo/1 → 201 {id:5, codigo:"POI-2.06"}
    4. POST /api/necesidades-pap → 201 {id:10, nombre:"Microscopios"}
    5. POST /api/expedientes → 201 {codigo:"EXP-2026-0001", estado:"Borrador"}
    6. PUT /api/expedientes/1/estado → 200 {estado:"En_revision"}
    7. PUT /api/expedientes/1/estado → 200 {estado:"Aprobado"}
    8. GET /api/notificaciones → 200 [notificaciones creadas automáticamente]
  Assert: todos los pasos retornan 2xx, datos consistentes entre servicios

Prueba 2 — Circuit Breaker (Tolerancia a Fallos):
  Herramienta: JUnit + Mockito
  Escenario: presupuesto-service DETENIDO
  Acción: POST /api/expedientes (intenta validar saldo)
  Resultado esperado:
    - Intento 1: falla → retry (500ms)
    - Intento 2: falla → retry (1000ms)
    - Intento 3: falla → Circuit Breaker OPEN
    - Respuesta al cliente: 503 {error:"Servicio de presupuesto no disponible. Intente más tarde."}
    - expediente-service NO crashea, sigue respondiendo otros endpoints

Prueba 3 — Eventos RabbitMQ (Integridad de Mensajería):
  Herramienta: RabbitMQ Management + Testcontainers
  Escenario: publicar 100 eventos "expediente.aprobado"
  Verificaciones:
    - Los 100 mensajes aparecen en la cola "expediente.eventos"
    - notificacion-service consume los 100 sin pérdida
    - 100 notificaciones creadas en sisexp_notificaciones
    - Orden de procesamiento respeta timestamp

Prueba 4 — Health Check Global:
  Herramienta: curl / Actuator
  Verificaciones:
    - GET /actuator/health en cada servicio → {"status":"UP"}
    - Eureka Dashboard → 4 servicios + Gateway UP
    - Gateway health → agrega información de todos los downstreams

SECCIÓN 3 — OPTIMIZACIONES PROPUESTAS
═══════════════════════════════════════════

Título: "Mejoras de Rendimiento y Escalabilidad"

Tabla de optimizaciones:

| Optimización | Problema que resuelve | Implementación | Impacto |
|---|---|---|---|
| Redis Cache compartido | Consultas repetitivas (lista POI por techo) | @Cacheable en métodos Feign | -40% latencia en lecturas |
| GZip en Gateway | Payloads JSON grandes (>1KB) | spring.cloud.gateway.filter.compress | -60% tamaño de respuesta |
| Conexiones HikariCP | Exceso de conexiones a DB | spring.datasource.hikari.maximum-pool-size=10 por servicio | Evita saturación PostgreSQL |
| Lazy Loading + DTOs | Sobre-fetching de datos | @OneToMany(fetch=LAZY) + Projection DTOs | -30% tamaño de payload |
| Índices DB por servicio | Búsquedas lentas | CREATE INDEX en cada DB replicando índices del monolito | -50% tiempo de queries |
| Rate Limiting en Gateway | Abuso de API | RequestRateLimiter con Redis | Protege servicios backend |
