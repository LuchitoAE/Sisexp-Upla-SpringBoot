SECCIÓN 2 — METODOLOGÍA 12-FACTOR APP
═══════════════════════════════════════════

Título: "Los 12 Factores — Aplicados a Cada Microservicio"

Presentar como rueda/círculo con 12 segmentos:

I. Codebase — Un repositorio Git con estructura multi-módulo Maven:
    sisexp-microservicios/
    ├── auth-service/pom.xml
    ├── presupuesto-service/pom.xml
    ├── expediente-service/pom.xml
    ├── notificacion-service/pom.xml
    ├── gateway-service/pom.xml
    └── pom.xml (parent)

II. Dependencias — Cada servicio declara solo las suyas en su pom.xml.
    Sin dependencias ocultas.

III. Configuración — Variables de entorno, NO hardcodeadas:
    SPRING_DATASOURCE_URL, JWT_SECRET, RABBITMQ_HOST...
    Spring Cloud Config Server como fuente centralizada.

IV. Backing Services — PostgreSQL, RabbitMQ, Eureka tratados como 
    recursos externos conectables vía URL.

V. Build/Release/Run — Separación estricta:
    Build (mvn package) → Release (docker build + tag) → Run (docker run)

VI. Procesos — Stateless. La sesión vive en el JWT del cliente, 
    no en memoria del servidor.

VII. Port Binding — Cada servicio expone su puerto:
    auth-service:8081, presupuesto:8082, expediente:8083, notif:8084

VIII. Concurrencia — Escalado horizontal:
    docker-compose up --scale expediente-service=3

IX. Desechabilidad — Inicio rápido, apagado graceful:
    Spring Boot graceful shutdown con timeout 30s

X. Paridad Dev/Prod — Mismo Dockerfile y docker-compose en local y Railway

XI. Logs — stdout/stderr como flujo continuo, sin archivos de log internos.
    Railway recoge y centraliza.

XII. Admin — Tareas administrativas como procesos one-off:
    DataInitializer como CommandLineRunner en cada servicio

SECCIÓN 3 — PATRONES DE MICROSERVICIOS
═══════════════════════════════════════════

Título: "Patrones Arquitectónicos Implementados"

Tabla de 6 patrones:

| Patrón | Propósito | Implementación en SISEXP |
|---|---|---|
| Database per Service | Aislamiento de datos | 4 DBs independientes (auth, presupuesto, expediente, notif) |
| API Gateway | Punto único de entrada | Spring Cloud Gateway (rutas, auth, rate limit) |
| Saga (Coreografía) | Transacciones distribuidas | RabbitMQ: evento de compensación si falla reserva de saldo |
| CQRS | Separar lectura/escritura | Reportes consultan réplicas de solo lectura de cada servicio |
| Event Sourcing | Historial como eventos | Cada cambio de estado del expediente es un evento inmutable |
| Circuit Breaker | Tolerancia a fallos | Resilience4j: si presupuesto-service falla, expediente responde con mensaje controlado |

SECCIÓN 4 — CRITERIOS DE SELECCIÓN Y COMPARATIVA
═══════════════════════════════════════════

Título: "Monolito vs Microservicios — Matriz de Decisión"

Tabla comparativa final:

| Criterio | Monolito (Actual) | Microservicios (Propuesto) | Recomendación |
|---|---|---|---|
| Tamaño proyecto | 1 JAR, ~3000 líneas | 5 JARs, ~600 líneas c/u | Monolito suficiente para este alcance |
| Equipo | 1-2 desarrolladores | 3-5 desarrolladores | Microservicios requiere más coordinación |
| Tiempo desarrollo | 4 semanas | 8-10 semanas | Monolito más rápido para prototipo |
| Escalabilidad | Vertical (+RAM/CPU) | Horizontal (+réplicas) | Microservicios escala mejor |
| Debugging | Un solo proceso | Trazas distribuidas (Sleuth) | Monolito más fácil de depurar |
| Costo Railway | ~$5/mes (1 servicio) | ~$20/mes (5 servicios + RabbitMQ) | Monolito más económico |
| Resiliencia | Si cae, cae TODO | Degradación graceful | Microservicios más resiliente |
| Adopción académica | Ideal para aprender | Ideal para portafolio profesional | Ambos válidos según objetivo |

Conclusión (recuadro destacado al pie):
"El monolito Spring Boot actual es la arquitectura correcta para el alcance 
académico del proyecto. La migración a microservicios con Spring Cloud sería 
el paso evolutivo natural si SISEXP escalara a producción real con múltiples 
facultades, mayor carga y equipos distribuidos."
